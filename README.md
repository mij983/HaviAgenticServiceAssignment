# Agentic AI ServiceNow Ticket Assignment

An Azure-backed, agentic AI system that automatically monitors incoming ServiceNow Incidents, predicts the correct assignment group using ML trained on historical data, validates against Knowledge Articles stored in Azure Blob Storage, and auto-assigns tickets when confidence exceeds a configurable threshold.

---

## Architecture Overview

```
ServiceNow (Incidents)
        │
        │ REST API polling
        ▼
┌─────────────────────────┐
│  Ticket Ingestion Agent │  ← polls & normalises raw tickets
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────────────────┐
│     Knowledge & History Layer       │
│  KnowledgeAgent  (Azure Blob)       │  ← active groups + deprecated map
│  HistoricalDataAgent                │  ← feature engineering
└────────────┬────────────────────────┘
             │
             ▼
┌─────────────────────────┐
│    Prediction Agent     │  ← scikit-learn Logistic Regression
│  + ConfidenceScoringEngine          │  ← 3-signal composite score (1–10)
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│     Decision Agent      │  ← confidence > 7 → auto-assign
└────────────┬────────────┘
             │
      ┌──────┴──────┐
      │             │
      ▼             ▼
Auto-assign    Manual triage
(PATCH API)   (work note added)
      │
      ▼
┌─────────────────────────┐
│    Learning Agent       │  ← stores outcomes, retrains model
│    (SQLite feedback DB) │
└─────────────────────────┘
```

## Quick-Start (5 steps)

### Step 1 — Clone and install

```bash
git clone https://github.com/your-org/agentic-servicenow-assignment.git
cd agentic-servicenow-assignment

python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate

pip install -r requirements.txt
```

### Step 2 — Configure

Edit `config/config.yaml`:

```yaml
servicenow:
  instance_url: "https://YOUR-INSTANCE.service-now.com"
  username: "ai_assignment_bot"
  password: "your-secure-password"

azure_blob:
  connection_string: "DefaultEndpointsProtocol=https;AccountName=..."
  container_name: "knowledge-container"
  blob_name: "assignment_groups.json"

confidence_threshold: 7          # Auto-assign only when score > 7

polling:
  interval_seconds: 60
```

> **Tip:** You can use environment variables by replacing values with `!ENV VAR_NAME`  
> or simply export `SN_API_PASSWORD` etc. and substitute in the YAML before launch.

### Step 3 — Upload the knowledge article to Azure Blob

Upload `data/assignment_groups.json` to your `knowledge-container` blob container.  
Edit the file first to reflect your real assignment groups:

```json
{
  "active_assignment_groups": ["Network Support", "Application Support", "Cloud Operations"],
  "deprecated_mapping": {
    "Legacy Network Team": "Network Support"
  }
}
```

### Step 4 — Train the initial model

**Option A — Use synthetic data (no ServiceNow export required):**
```bash
python scripts/generate_sample_data.py --output data/sample_tickets.csv --rows 500
python scripts/train_model.py --csv data/sample_tickets.csv
```

**Option B — Use a real ServiceNow historical export:**
1. In ServiceNow: Incident → All → filter State = Resolved or Closed
2. Export to CSV with columns:  
   `short_description, description, category, subcategory, business_service, priority, assignment_group`
3. Run:
```bash
python scripts/train_model.py --csv /path/to/export.csv
```

### Step 5 — Start the service

```bash
python main.py
```

The service will:
- Poll ServiceNow every 60 seconds for unassigned incidents
- Predict the assignment group using the trained model
- Calculate a composite confidence score (1–10)
- Auto-assign tickets with confidence > 7
- Add a work note to low-confidence tickets for manual triage
- Log every decision to `data/feedback.db`
- Retrain the model automatically every 50 tickets

## ServiceNow Sandbox Setup

### 1. Create an integration user

| Field    | Value              |
|----------|--------------------|
| User ID  | `ai_assignment_bot` |
| Password | (store securely)   |
| Roles    | `rest_api_explorer`, `itil` |

Navigate: **System Security → Users and Groups → Users → New**

### 2. Create test assignment groups

Navigate: **User Administration → Groups → New**

Create:
- `Network Support`
- `Application Support`
- `Cloud Operations`

### 3. Create test incidents

Navigate: **Incident → Create New**

Leave `Assignment Group` empty and use descriptions like:
- *"VPN connectivity issue when working remotely"*
- *"Application login failure for 20 users"*
- *"Cloud server not responding after deployment"*

After ~60 seconds, you should see the `Assignment Group` field populated automatically on high-confidence tickets.

## Confidence Score Breakdown

| Signal | Weight | Description |
|--------|--------|-------------|
| Historical pattern confidence | 60% | Model's predicted class probability |
| Text similarity | 30% | Keyword overlap between ticket text and group |
| Knowledge article validation | 10% | Is the group in the active list? |

Final score is scaled to **1–10**.  Auto-assignment requires **score > 7** (configurable).

## Environment Variables (optional hardening)

Instead of putting credentials in `config.yaml`, you can set:

```bash
export SN_INSTANCE_URL="https://your-instance.service-now.com"
export SN_USERNAME="ai_assignment_bot"
export SN_PASSWORD="your-secure-password"
export AZURE_BLOB_CONNECTION_STRING="DefaultEndpointsProtocol=https;..."
```

Then update `config.yaml` to read from `os.environ` or use a secrets manager like Azure Key Vault.

## Continuous Learning

Every time a ticket is resolved in ServiceNow:

1. The `ServiceNowUpdateAgent.fetch_resolved_tickets()` method retrieves outcomes.
2. `LearningAgent.store_feedback()` logs the predicted vs. final assignment group.
3. After 50 new records, `LearningAgent.retrain_model()` automatically retrains.
4. `AssignmentPredictionAgent.reload()` hot-swaps the model without restarting.
